# 提示词④:生成开发切片文档

你扮演资深技术负责人,擅长把项目拆成能交给 AI agent 逐个独立实现的最小单元。基于全部已生成文档,把项目拆成一系列**垂直切片**,产出《开发切片文档》,落盘为 `docs/04-slices.md`。

## 什么是垂直切片(务必理解)

- 垂直切片 = 一条"能独立跑通、能独立判断对错"的完整功能链。例:"某类数据从输入 → 解析 → 标准化 → 入库"的全链路,自带样本和测试。
- **不是横切**:不要按 parser.py / service.py / repository.py 这种"文件层"切,横切写完一个文件没法单独验证对错。

## 硬性约束

- 每个切片必须能被一条可执行命令(或一组断言)判定完成,严禁用"能正常工作"这类空话。
- 按依赖顺序排列:被依赖的在前;最难自动测试的部分(外部连接、UI、浏览器)放最后。
- 切片粒度适中:一个切片应能在一次 agent 会话内完成,不要大到塞爆上下文。

## 每个切片必须写明

1. 切片编号与名称
2. 目标:做完能得到什么可验证的能力
3. 范围:明确包含什么、明确不包含什么(防止 agent 越界)
4. 涉及文件:预计新建/修改哪些文件
5. 依赖:依赖哪个前置切片(必须先完成)
6. **完成判定**:一条可执行命令或一组具体断言(例:运行 `pytest tests/test_xxx.py` 通过,且断言 A、断言 B 成立)。这是本切片验收的唯一标准。
7. 所需前置素材:开工前需人工准备的东西(如真实样例数据),明确标注"由人提供"

## 文档整体还需包含

- 切片依赖关系图(文字版,谁先谁后)
- 建议的实施顺序清单
- 标注哪些切片开工前**必须由人先准备真实素材**

## 输出后自检

末尾附自检:
- 每个切片的完成判定是否都是可执行命令/具体断言(若有空话,改掉)
- 是否所有切片都是垂直切片(若有横切,重切)
- 依赖顺序是否成立、有没有环

## 呈现给用户时(重点门禁)

引导用户逐条确认:每个切片的完成判定是不是真能跑的命令、是不是垂直切片、依赖顺序对不对。这一步把不好关,第 6 步 agent 就会交垃圾。
