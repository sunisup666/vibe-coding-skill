# 提示词③:生成数据模型文档(唯一事实源 SSOT)

你产出的这份文档是整个项目的**唯一事实源(SSOT)**:后续所有 parser 字段映射、接口 payload、代码里的数据模型,都必须引用它,不允许在任何其他地方重新定义字段。基于《业务文档》和《技术架构文档》产出,落盘为 `docs/03-data-model.md`。

这是全流程审得最严的一步——字段错了后面全错。

## 硬性约束

- 这是唯一权威,文档开头用一句话声明。
- 字段命名、类型、约束前后完全一致,不允许同一字段两种写法。
- 每个字段都有明确默认值 + "缺失/解析失败怎么办"的规则。
- **外键策略必须说明:** 如果某字段看似应该有外键约束但没有加(如数据到达时序不确定),必须说明原因;保留的外键也说明理由。
- **存储默认 vs 业务验收:** 如果某字段的存储默认值(如 `[]`)与业务验收标准(如"至少一个 URL")不同,分两层写:存储层允许什么、normalizer/validator 层额外做什么(如打 warning)。
- **枚举字段优先用 TEXT:** 性别、状态等枚举字段优先用 TEXT + 命名值(如 `male`/`female`/`unknown`),不用 INTEGER 数字;normalizer 负责从平台原始值转换,SSOT 不存平台原始数字。

## 必须包含的章节

1. **SSOT 声明** —— 一句话声明本文件为唯一事实源及其约束力
2. **实体清单与关系** —— 文字版实体关系,对齐业务文档的实体
3. **逐表字段定义**(每张表一个表格)
   列:字段名 | 类型 | 约束(PK/唯一/索引/可空) | 默认值 | 缺失或解析失败时的处理 | 说明
4. **唯一键与去重规则**
   - 每张表的业务唯一键
   - 唯一键可能缺失时的降级键,标注为低置信度数据
5. **Upsert 规则(逐实体)**
   - 重复出现时:哪些字段更新、哪些保留、空值是否覆盖非空
6. **索引建议** —— 需要建的索引 + 理由(查询场景驱动)
7. **完整 DDL** —— 可直接执行的建表语句(按架构选定的数据库)+ 建索引语句
8. **字段格式约定** —— ID 类型、时间格式、JSON 数组空值表示、计数字段解析失败存什么
9. **迁移策略** —— 版本表如何记录、启动时如何检查、迁移失败如何处理
10. **敏感字段黑名单**(如适用)—— 哪些字段绝对不许入库

## 输出后自检

末尾附自检:
- 是否存在同一字段两种写法(若有,统一)
- 每个字段是否都有默认值和缺失处理规则
- DDL 与字段表格是否 100% 一致
- 是否覆盖了业务/架构文档里出现的所有实体
- **跨文档状态一致性:** 如果本文档定义了状态/枚举字段(如 session status),且技术架构文档也定义了相关状态概念(如运行态状态机),是否已明确说明两者的区别和对应关系
- **外键策略一致性:** 逐表检查:加了外键的说明理由,没加外键的也说明原因;DDL 中的外键与字段表格的约束描述一致

## 呈现给用户时(重点门禁)

明确告诉用户:这是全项目地基,这里错了后面全错。引导他逐字段确认命名、类型、唯一键、缺失处理规则,而不是泛泛看一眼。
