# 常见翻车点检查清单

在把任何生成的文档呈现给用户之前,对照本清单相关项自查一遍,发现问题当场修掉。这是在用户之前拦住低质量文档的质量门。

## 全流程通用

| 翻车点 | 后果 | 正确做法 |
| --- | --- | --- |
| 一次生成全部文档 | 用户无法逐步审阅,反馈难聚焦 | 一次只推进一步 |
| 文档里堆空话("提升效率""优化体验") | 无法落地、无法验证 | 每条都要具体、可判断真假 |
| 跳步生成(前置文档还没定稿就往后做) | 后面的文档建立在流沙上 | 前置未定稿不进入下一步 |

## 第 1 步:业务文档

- [ ] 有没有混入任何技术实现细节(框架、字段类型、表结构、API)?有则删除,业务文档只讲业务。但"本地存储""URL"等工程边界词汇如果是业务可验收的必要保留,标注后保留。
- [ ] 每条业务规则是否都可判断真假?模棱两可的要改具体。
- [ ] "不做事项"是否明确?范围边界不清后面必然蔓延。
- [ ] 核心业务实体是否覆盖齐全?后续数据模型要依赖它。
- [ ] **有没有为中间数据/摘要数据单独建实体?** 如果某数据是其他实体的摘要/来源,考虑作为来源数据参与 upsert 而非独立建模。
- [ ] **验收视角中的每一条是否具体到可测试?** "视频信息"这种模糊表述应改为"video_urls 数组至少一个有效 URL"。

## 第 2 步:技术架构文档

- [ ] 是否覆盖了业务文档里的每个实体和主流程?有遗漏要补。
- [ ] 有没有为了"显得专业"引入的过度设计?能简单就简单。
- [ ] 依赖外部/难测试的部分,是否都被隔离成了可离线单测的薄接口?
- [ ] 技术选型是否都给了一句话理由?
- [ ] **依赖方向是否自洽?** "禁止依赖"列表与"依赖总览"不能矛盾;编排层负责跨层组装,接入层不直接调数据层。
- [ ] **跨文档状态/枚举一致性?** 本文档的状态概念与业务文档的同类概念是否一致或已说明差异。
- [ ] **第三方库平台风险是否标注?** 不声称"跨平台"除非已验证。
- [ ] **策略是否定稿?** 有多个可选方案时选了一个并说明理由,没有"可以A也可以B"。

## 第 3 步:数据模型文档(SSOT,审得最严)

- [ ] **同一字段有没有两种写法**?命名、类型必须全项目一致。
- [ ] **每个字段是否都有默认值 + 缺失/解析失败时的处理规则**?
- [ ] DDL 与上面的字段表格是否 100% 一致?
- [ ] 每张表的业务唯一键是否明确?唯一键可能缺失时有没有降级键?
- [ ] Upsert 规则是否写清了"哪些字段更新、哪些保留、空值是否覆盖非空"?
- [ ] 是否覆盖了业务/架构文档里出现的所有实体?
- [ ] 开头是否声明了本文件为唯一事实源?
- [ ] **外键策略是否说明?** 加了外键的说明理由,没加的(如数据到达时序不确定)也说明原因。
- [ ] **存储默认 vs 业务验收是否分清?** 如存储允许 `[]` 但业务要求"至少一个 URL",是否分两层写了存储允许什么 + normalizer 做什么。
- [ ] **枚举字段是否用 TEXT 而非 INTEGER?** 避免不同平台数字映射冲突。
- [ ] **跨文档状态一致性?** 数据模型的 status 字段与技术架构的状态机是否已说明区别和对应关系。

## 第 4 步:开发切片文档(审完成判定)

- [ ] **每个切片的完成判定是不是可执行命令 / 具体断言**?出现"能正常工作""功能完整"这类空话一律改掉。
- [ ] **是不是垂直切片**(一条完整功能链),而不是按 parser.py / service.py 横切?横切的重切。
- [ ] 依赖顺序是否成立、有没有环?最难自动测试的 I/O(外部连接、UI)是否排在最后?
- [ ] 每个切片是否标明了"开工前需人工准备的素材"(如真实样例)?
- [ ] 切片粒度是否适中(一次 agent 会话内能完成,不塞爆上下文)?
- [ ] **文档开头是否声明了切片总数?** 正文切片数与呈现给用户时的口径必须一致。
- [ ] **涉及文件列表是否包含依赖配置变更?** 引入新库时必须列出 `pyproject.toml`/`package.json` 等修改。
- [ ] **切片间文件边界是否清晰?** 后序切片修改前序切片创建的文件时,前序切片是否明确了只做了什么、留了什么。
- [ ] **fixture 是否按用例拆分?** 需要覆盖多种情况的素材是否拆成独立文件和独立测试函数。
- [ ] **完成判定粒度是否满足下游?** 下游切片依赖的输出格式(如统计格式)是否在本切片验收中验证。
- [ ] **依赖三处一致?** 依赖关系图、各切片"依赖"字段、自检说明中的依赖链不能矛盾。
- [ ] **端到端切片是否包含敏感字段验证?** 最终集成测试必须有安全红线的端到端断言。

## 第 5 步:AGENTS.md

- [ ] 禁止事项是否都从前面文档如实提炼、无遗漏?
- [ ] 是否声明了数据模型文档为唯一字段权威?
- [ ] 验证命令是否具体可执行?
- [ ] "何时必须停下问人"是否覆盖了 agent 容易擅自猜测的场景(未定义字段、需要真实数据、需求歧义、引入新依赖)?

## 第 6 步:逐切片实现

- [ ] 是否只做了当前指派的切片,没顺手改别的?
- [ ] 是否严格对齐数据模型,没自创字段?
- [ ] 是否先写测试再写实现?
- [ ] 验证命令是否真的跑了、真的全绿(而不是嘴上说通过)?
- [ ] 依赖外部真实数据的切片,用户是否已经提供了样例?没提供就该停下来要,而不是猜着写。
