# 提示词复盘与回流方法

本文件定义:一份文档定稿后,如何复盘对应提示词、只提炼可泛化改进、并回流到模板。这是本 skill 的自优化机制。做提示词复盘前读本文件。

## 何时触发

某一步的文档经过审阅迭代、用户明确表示满意、已落盘之后。

## 复盘的输入

复盘的原材料是这一步产生的"改动轨迹":
1. 提示词原本要求了什么。
2. 第一版草稿产出了什么。
3. 用户在审阅循环里提了哪些反馈。
4. 定稿相对初稿改了哪些地方。

用户反馈修正掉的东西,恰恰暴露了提示词没说清的地方。这就是要改进的靶子。

## 核心红线:只提炼"可泛化缺陷",不搬"项目特有内容"

这是整个机制成败的关键。判别标准:

**可泛化缺陷(应回流进提示词)** —— 换一个项目也会犯的同类毛病。例如:
- "初稿字段没写默认值,用户每次都得补" → 提示词加一条:每个字段必须给默认值和缺失处理。
- "初稿切片完成判定写成'能正常工作',用户要求改成具体命令" → 提示词强化:完成判定必须是可执行命令。
- "初稿总把技术细节混进业务文档" → 提示词加一条更明确的禁止项 + 自检。
- "章节结构缺了术语表,导致后续文档术语不统一" → 提示词补上该章节要求。

**项目特有内容(绝不回流)** —— 只属于这个项目的具体信息。例如:
- "用户要求给笔记表加一个 video_duration 字段" → 这是本项目的字段,只进本项目的数据模型文档,不进提示词。
- "把评论去重键改成 note_id+user_id" → 本项目的业务规则,不进提示词。
- 任何具体的表名、字段名、业务实体、领域术语。

**一句话判别:** 这条改进,拿去做一个完全不同领域的项目还成立吗?成立 → 可泛化,回流;不成立 → 项目特有,不动提示词。

判断不准时,宁可不改,并问用户一句:"这条我拿不准算不算通用改进,你觉得要不要写进提示词模板?"

## 回流的动作

1. 找出 1~3 条可泛化改进(通常不会更多;没有就明确说"这步提示词这次表现良好,无需改动")。
2. 对提示词做**最小必要修改**:加一条约束 / 补一个章节要求 / 加一个反例 / 收紧一句含糊表述。不要大改重写。
3. 给用户看 before/after 的 diff,说明每条改动对应哪次反馈。
4. 用户确认后,写回当前项目 `prompts/` 对应文件。

## 回流:写回本 skill 自己

提示词只有一份,就在本 skill 的 `references/prompts/`。可泛化改进直接写这里——全局安装是 `git clone` 的可写仓库。当前项目后续步骤立刻受益,其它项目下次运行也受益。

整个项目走完(或用户要求)时,汇总本次改动,**给用户看 diff,用户确认后**在 skill 目录 `git commit && git push` 推到 GitHub。不静默自动推送:一次糟糕的改动推上去会污染之后所有项目。

`SKILL.md`(引擎)不在自动回流范围内,只手动改。

## 反模式(不要这么做)

- 不要每步都强行改提示词凑数。没有可泛化缺陷就不改。
- 不要把项目业务内容写进提示词(见红线)。
- 不要大段重写提示词——那会丢掉原本已经打磨好的部分,只做增量修改。
- 不要不给用户看 diff 就偷偷改——回流必须经用户确认。
- 不要把自检写成结论式背书("已确认""均一致")。回流时如果发现某类缺陷明明有对应检查项却仍漏过,别急着加新检查项——那说明问题在自检执行不够较真,应改"呈现前自检"的执行方式(要求逐条引用原文佐证),而非往清单里堆条目。
