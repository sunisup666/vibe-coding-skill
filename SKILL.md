# Vibe Coding 工程化流水线

把"脑子里的项目想法"翻译成一条**文档链**,再交给 agent 逐切片实现。用户只负责三件事:**写/审文档、定边界、审关键点**,不碰实现代码。本 skill 负责一步步引导、每步审阅迭代、并在满意后自动优化本阶段提示词并回流母版。

## 核心理念:三条铁律(贯穿全流程)

1. **唯一事实源(SSOT)**:数据模型文档(`docs/03-*.md`)是唯一字段权威。字段只在这里定义一次,别处只引用不重写。字段散落多份文档 = 必然漂移。
2. **完成判定必须是能跑的命令**:"能解析视频"不算完成,"运行 `pytest tests/test_video.py` 通过"才算。没有可执行门禁,agent 会交"看着对"的垃圾,用户还得亲自读代码。
3. **垂直切片,不是横切**:交给 agent 时一次只给一条"能独立跑通、独立判对错的完整功能链",不是按文件横切。

在整个引导过程中,任何生成的文档如果违反这三条,必须在呈现给用户之前自己先修掉。

## 开始前:定位工作区并判断进度

1. 查看当前目录是否是从 project-template fork 出来的项目,应包含:`prompts/`、`skeleton/`、`docs/`、`AGENTS.md`。
2. 如果目录里**没有** `prompts/`,说明用户还没铺骨架。询问是否要用本 skill 自带的模板初始化:把 `references/prompts/` 复制成项目的 `prompts/`,创建空的 `docs/`。同意后再做。
3. 如果 `docs/` 里**已经有部分文档**,说明用户是回来续做的。读一遍已有文档,判断进行到第几步,从下一步继续,不要重头再来。
4. 判断用户意图是**全新项目**还是**改造已有项目**。改造项目时,先让用户提供或指出现有代码/文档,把它作为概念输入,业务文档里要明确"现状 → 目标"的差异。

确认清楚"用户在流水线的哪一步"再开始推进。**一次只推进一步**,不要一口气把所有文档都生成出来。

## 流水线总览

| 步骤 | 产物文件 | 用的提示词 | 人工门禁 |
| --- | --- | --- | --- |
| 0 | (对话中形成的概念一页纸) | `prompts/00-concept-helper.md` | 用户确认概念 |
| 1 | `docs/01-business.md` | `prompts/01-business-doc.md` | 审阅 |
| 2 | `docs/02-architecture.md` | `prompts/02-tech-architecture.md` | 审阅 |
| 3 | `docs/03-data-model.md`(**SSOT**) | `prompts/03-data-model.md` | **重点审阅** |
| 4 | `docs/04-slices.md` | `prompts/04-dev-slices.md` | **审完成判定** |
| 5 | `AGENTS.md` | `prompts/05-agents-md.md` | 审阅 |
| 6 | 各切片实现代码 | `prompts/06-slice-implement.md` | **每切片审绿灯** |

**提示词来源规则**:优先用当前项目 `prompts/` 里的版本(可能已被前几个项目优化过);当前项目没有该文件时,回退到本 skill 的 `references/prompts/`。

## 每一步的标准循环(1~5 步都走这个循环)

对每一个文档步骤,严格按下面顺序执行:

### 1) 读提示词 + 收集输入
读取该步的提示词文件,以及它需要的所有前置文档(如第 3 步需要业务文档 + 架构文档)。前置文档缺失就先回去补,不要跳步。

### 2) 生成草稿
按提示词的要求生成文档草稿。**不写代码,只写该阶段该有的内容。**

### 3) 呈现前自检(拦住翻车)
在给用户看之前,先对照 `references/pitfalls.md` 里与本步相关的检查项自查一遍,发现问题当场修掉。尤其:
- 第 1 步:有没有混入技术实现细节。
- 第 3 步:同一字段有没有两种写法;每个字段是否都有默认值和缺失处理;DDL 与字段表是否一致。
- 第 4 步:每个切片的完成判定是不是可执行命令(不是"能正常工作"这种空话);是不是垂直切片而非横切。

### 4) 呈现 + 明确告诉用户"这一步重点看什么"
把文档给用户,并**主动指出这一步他应该重点审的地方**(见"人工门禁")。不要只说"你看看",要引导他看关键处。例如第 3 步:"这是全项目地基,请重点确认字段命名、唯一键、缺失字段的处理规则,这里错了后面全错。"

### 5) 审阅迭代循环
- 用户满意 → 落盘到对应 `docs/` 文件,进入第 6 小步。
- 用户不满意 → 记录他的每条反馈,修改后再次呈现。**不要一次问一堆问题**,顺着他的反馈一点点改。反复,直到用户明确表示满意为止。

### 6) 提示词复盘与回流(满意之后才做)
文档定稿后,按 `references/prompt-tuning.md` 的方法做一次复盘:对比"初稿 → 定稿"之间因用户反馈产生的改动,判断哪些是**可泛化的提示词缺陷**(而不是本项目特有内容),把可泛化的部分改进到本步提示词里,给用户看 before/after diff,确认后写回当前项目 `prompts/` 对应文件。

**关键红线:只回流可泛化的改进,绝不把某个项目的具体业务内容塞进提示词。** 区分方法见 `references/prompt-tuning.md`。判断不准时,宁可不改,也问用户一句。

### 7) 进入下一步
简短说明"这一步完成了什么、接下来第 X 步要做什么、需要用户先准备什么(如有)",再继续。

## 人工门禁(必须停下来提醒用户重点审的三处)

- **第 3 步之后(数据模型)**:全项目地基,字段错了后面全错。引导用户逐字段确认命名、类型、唯一键、缺失处理。这里审得最细。
- **第 4 步之后(开发切片)**:逐条确认每个切片的"完成判定"都是能跑的命令或具体断言;确认都是垂直切片。这里没把好关,第 6 步 agent 就会交垃圾。
- **第 6 步每个切片之后**:只看绿灯是否通过 + 难以自动测试的边缘(外部连接、UI、浏览器)。

## 开工前硬前提(依赖外部真实数据的项目)

如果项目依赖外部真实数据(爬取、第三方接口响应、外部文件格式),在进入第 6 步前,**必须让用户亲手准备几组"真实输入 + 期望输出"样例**。这是 agent 猜不出来的,也正是用户该负责的"上下文之外"的事。样例不齐,不进第 6 步——明确告诉用户这一点。

## 第 6 步:逐切片实现(把活交给 agent)

这一步不同于前面。按 `docs/04-slices.md` 的顺序,**一次只实现一个切片**:

1. 读 `prompts/06-slice-implement.md`,连同全部文档 + `AGENTS.md` 作为上下文。
2. 只做当前切片划定的范围,严格对齐 `docs/03-data-model.md`,禁止自创字段。
3. 先写测试,再写实现(TDD)。
4. 跑该切片规定的验证命令,必须全绿,把结果给用户看。
5. 遇到文档未定义的结构、需要真实样例、需求有歧义、要引入新依赖时,停下来问用户,不许自己猜。
6. 用户确认绿灯 → 下一个切片。

## 项目收尾:回流母版模板

一个项目全部走完后:
1. 询问用户母版 project-template 仓库的路径。
2. 把本项目 `prompts/` 里被优化过的提示词,与母版对比,把可泛化的改进同步回母版(用户确认后再写)。
3. 给一份简短复盘:哪几步的提示词最容易生成低质量文档、这次改了什么、下次可注意什么。

这样跑三四个项目后,母版里的提示词会越来越准——这是本 skill 最终沉淀的资产。

## 沟通风格

- 一次只推进一步,清楚交代"现在在哪、接下来干什么、你要审什么"。
- 用户是技术人,直接、不绕弯、不堆废话。
- 呈现文档时主动指出重点审查项,别让用户自己猜该看哪。

## 参考文件

- `references/prompts/00-06*.md` — 每一步的详细提示词(生成对应文档的完整规格)。执行到某一步时读取对应文件。
- `references/pitfalls.md` — 常见翻车点检查清单。每步"呈现前自检"时对照。
- `references/prompt-tuning.md` — 提示词复盘与回流的具体方法,含"可泛化 vs 项目特有"的判别标准。做第 6 小步(提示词复盘)前读它。
